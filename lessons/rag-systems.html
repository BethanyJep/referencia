📖 Introduction: Whiskers Goes to the Library

You ask Whiskers 🐱: "What happened in ancient Egypt?" Two things can happen.

Option A — No Library: Whiskers puffs up confidently and announces: "Cleopatra invented pizza and the pyramids were built as giant cat toys!" 😬 This is called a hallucination — the model confidently makes up plausible-sounding facts it doesn't actually know. This happens when an LLM's training data doesn't contain the answer or when its knowledge cut-off predates recent events.

Option B — RAG: Before answering, Whiskers pads over to a carefully organised library 📚, searches the shelves for the relevant pages about ancient Egypt, reads them, and then answers — grounded in the actual text. The answer is accurate, citable, and up to date. That's Retrieval-Augmented Generation (RAG)!

RAG solves three of the biggest pain points with vanilla LLMs: outdated knowledge, no access to private data, and confident hallucinations. It is now the dominant architecture for enterprise AI applications.

💡 Key Insight

RAG = Retrieval + Generation. Instead of relying on memorised training data, the model dynamically fetches relevant context from a knowledge base and uses it to produce a grounded, accurate answer.

"Give the LLM a reference library to consult before it speaks!"

📚 Core Content

Retrieval-Augmented Generation

RAG is an AI architecture that combines a retrieval system (finding relevant information) with a generative model (producing text). Instead of relying purely on knowledge baked in during training, the model dynamically looks things up at inference time.

Published by Meta AI researchers in 2020, RAG has become the default architecture for knowledge-intensive applications.

🤔 Why RAG Exists: The Three Problems It Solves

🕰️ Knowledge Cutoff

LLMs only know what was in their training data. RAG lets them access current, live information.

🔒 Private Data

Your company's internal documents were never in GPT's training data. RAG gives the model access to them.

😬 Hallucinations

LLMs confabulate when uncertain. Grounding the answer in retrieved text dramatically reduces made-up content.

🐱 The Three RAG Steps

  1. Retrieve: Embed the user's question as a vector → search a vector database → fetch the top-K most relevant document chunks
  2. Augment: Prepend the retrieved chunks to the LLM prompt as context: "Use the following information to answer: [chunks] \n\nQuestion: [user query]"
  3. Generate: The LLM reads the context and produces a grounded, accurate answer

💼 Classic Use Case: Company Chatbot

Build a support bot that can answer questions about your specific company:

  • Index your product docs, FAQs, support articles, and internal policies
  • When a customer asks a question, retrieve the 3 most relevant chunks
  • Feed those chunks to GPT-4 along with the question
  • Get an accurate, grounded answer — no hallucinations about your refund policy!

Without RAG: the LLM doesn't know your company exists. With RAG: it's an expert on your product! 🎯

The RAG pipeline has two distinct phases that run at different times:

🏗️ Phase 1: Indexing (offline)

Done once (or when docs change)

1️⃣ Load Documents

PDFs, Word files, web pages, databases, Notion pages...

2️⃣ Chunk

Split into smaller pieces (~512 tokens each)

3️⃣ Embed

Convert each chunk to a vector using an embedding model

4️⃣ Store

Save vectors + text in a vector database

🔎 Phase 2: Querying (live)

Runs on every user question

1️⃣ Embed Query

Convert user's question to a vector

2️⃣ Retrieve

Find the top-K most similar chunks in the DB

3️⃣ Augment Prompt

Inject retrieved chunks as context before the question

4️⃣ Generate

LLM produces a grounded answer using the context

📝 What Goes in the Augmented Prompt?

System: You are Whiskers, a helpful cat assistant. Answer only based on
the provided context. If the answer isn't in the context, say so.

Context:
---
[Chunk 1]: "Our refund policy allows returns within 14 days of purchase,
with original receipt. Items must be in original condition..."

[Chunk 2]: "Digital downloads are non-refundable once accessed. For
subscription products, please see section 4.2..."
---

User Question: How long do I have to return a product?

The LLM now has the facts it needs — no guessing required! 🎯

What is a Vector Database?

A vector database is a specialised database designed to store, index, and query high-dimensional embedding vectors at speed. Unlike a traditional SQL database that finds rows by exact ID or keyword match, a vector DB finds results by semantic similarity — asking "what vectors are closest to this query vector?"

Think of it as the library catalog that finds books by meaning and topic, not just by exact title match. Whiskers can ask for "books about ancient cat royalty" and get results even if none are titled exactly that. 📖🐱

🔍 How Vector Search Works (ANN)

Searching millions of vectors by brute force (comparing every vector) would be too slow. Vector DBs use Approximate Nearest Neighbour (ANN) algorithms like HNSW (Hierarchical Navigable Small World) to find the closest vectors in milliseconds.

The trade-off: results are approximate (might miss 1–2% of true matches) but the speed gain is enormous — from minutes to milliseconds at scale.

Why Chunking Matters

You can't embed an entire 100-page PDF as one vector — that vector would be a muddy average of everything, useless for precise retrieval. Instead, you split documents into chunks: smaller pieces that each encode one focused idea.

Think of chunking as Whiskers tearing a textbook into individual flash cards before filing them in the library. Each card covers one topic, making it easy to find exactly the right one later! 🐱📇

The Four Main Strategies

📏 Fixed-Size Chunking

Split every N characters or tokens. Simple and fast to implement. No linguistic awareness — may cut mid-sentence.

Good for: Quick prototypes, simple documents

chunk_size=500, overlap=50

📝 Sentence-Level

Split on sentence boundaries (periods, newlines). More natural — each chunk is a complete thought. Slightly more complex but much better quality.

Good for: General documents, FAQs

🌳 Recursive (Smart)

LangChain's default. Tries to split on paragraphs first, then sentences, then words — preserving as much logical structure as possible.

Good for: Most real-world documents

🧠 Semantic Chunking

Embed as you go, split when the embedding drifts significantly. Groups text by actual meaning change. Expensive but produces the best retrieval quality.

Good for: High-quality production RAG

⭐ Best Practices for Chunking

  • Chunk size: ~512 tokens works well for most use cases. Too small = no context. Too large = diluted retrieval.
  • Overlap: Add 50–100 token overlap between consecutive chunks so no critical information is lost at a boundary.
  • Metadata: Store the source filename and page number with each chunk so you can cite your sources.
  • Test your retrieval: Always test whether the right chunks get retrieved for your expected queries before deploying.
# pip install langchain langchain-community
from langchain.text_splitter import RecursiveCharacterTextSplitter

# Smart recursive chunking with overlap
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,           # target chunk size in characters
    chunk_overlap=50,         # overlap between consecutive chunks
    separators=["\n\n", "\n", ". ", " ", ""],  # try these in order
)

long_document = """
Whiskers the cat has many fascinating qualities.
She sleeps 14 hours a day and hunts moths at night.

The history of domesticated cats dates back 10,000 years.
Ancient Egyptians revered them as sacred animals, even
mummifying them as offerings to the goddess Bastet.
"""

chunks = splitter.split_text(long_document)
for i, chunk in enumerate(chunks):
    print(f"Chunk {i+1} ({len(chunk)} chars): {chunk[:80]}...")

💻 End-to-End RAG with LangChain + Chroma

# pip install langchain langchain-openai langchain-chroma chromadb
import os
from langchain_community.document_loaders import TextLoader, DirectoryLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_chroma import Chroma
from langchain.chains import RetrievalQA

# ── PHASE 1: INDEXING ─────────────────────────────────────────────────

# 1. Load documents (txt, pdf, etc. from a directory)
loader = DirectoryLoader("./cat_docs/", glob="**/*.txt", loader_cls=TextLoader)
documents = loader.load()
print(f"Loaded {len(documents)} document(s)")

# 2. Chunk the documents
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=50,
)
chunks = splitter.split_documents(documents)
print(f"Split into {len(chunks)} chunk(s)")

# 3. Embed and store in Chroma vector DB
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectordb = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db",  # persist to disk
)
print("Indexed and saved to Chroma!")

# ── PHASE 2: QUERYING ─────────────────────────────────────────────────

# Load existing DB (skip indexing on subsequent runs)
vectordb = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings,
)

# 4. Create a retriever (fetch top 3 relevant chunks)
retriever = vectordb.as_retriever(search_kwargs={"k": 3})

# 5. Build the RAG chain
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",       # "stuff" = concatenate all chunks into prompt
    retriever=retriever,
    return_source_documents=True,
)

# 6. Ask Whiskers anything! 🐱
query = "What do cats eat and how often should they be fed?"
result = qa_chain.invoke({"query": query})

print("\n🐱 Whiskers says:")
print(result["result"])
print("\n📄 Sources used:")
for doc in result["source_documents"]:
    print(f"  - {doc.metadata.get('source', 'unknown')}")

💻 RAG Without Frameworks (Raw Python)

# pip install openai chromadb
import chromadb
from openai import OpenAI

client = OpenAI()
chroma = chromadb.Client()
collection = chroma.create_collection("whiskers_library")

# Index some cat facts
cat_facts = [
    "Cats sleep between 12 and 16 hours per day.",
    "A cat's purring vibrates at 25–50 Hz, which promotes bone healing.",
    "Cats have 32 muscles in each ear, allowing precise directional hearing.",
    "The ancient Egyptians worshipped cats and mummified them as offerings.",
    "A group of cats is called a clowder; a group of kittens is a kindle.",
]

# Embed and store
embeddings_response = client.embeddings.create(
    model="text-embedding-3-small",
    input=cat_facts,
)
vectors = [e.embedding for e in embeddings_response.data]

collection.add(
    embeddings=vectors,
    documents=cat_facts,
    ids=[f"fact_{i}" for i in range(len(cat_facts))],
)

# Query: Retrieve + Generate
user_question = "How long do cats sleep?"

q_embed = client.embeddings.create(
    model="text-embedding-3-small", input=[user_question]
).data[0].embedding

results = collection.query(query_embeddings=[q_embed], n_results=2)
retrieved = "\n".join(results["documents"][0])

# Augmented prompt
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "system", "content": "Answer based only on the provided context."},
        {"role": "user", "content": f"Context:\n{retrieved}\n\nQuestion: {user_question}"},
    ],
)
print(response.choices[0].message.content)
# "Cats sleep between 12 and 16 hours per day." ✅🐱

💻 Mini Challenge: Whiskers' Cat Fact Checker

# Build a minimal RAG fact-checker in under 30 lines!
# pip install chromadb openai

import chromadb
from openai import OpenAI

client = OpenAI()
db = chromadb.Client()
col = db.create_collection("cat_facts")

# Your knowledge base — add as many facts as you like!
facts = [
    "Cats have retractable claws, unlike dogs whose claws are always exposed.",
    "A cat's nose print is unique, like a human fingerprint.",
    "Cats cannot taste sweetness — they lack the taste receptor for it.",
    "The average cat lives 12 to 18 years.",
    "Cats use their whiskers to gauge whether a space is wide enough to fit through.",
]

# Index
vecs = [e.embedding for e in client.embeddings.create(
    model="text-embedding-3-small", input=facts
).data]
col.add(embeddings=vecs, documents=facts, ids=[str(i) for i in range(len(facts))])

def ask_whiskers(question: str) -> str:
    q_vec = client.embeddings.create(
        model="text-embedding-3-small", input=[question]
    ).data[0].embedding
    top = col.query(query_embeddings=[q_vec], n_results=2)["documents"][0]
    context = "\n".join(top)
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": "You are Whiskers 🐱. Answer using only the context."},
            {"role": "user", "content": f"Context:\n{context}\n\nQ: {question}"},
        ],
    )
    return resp.choices[0].message.content

# Test it!
print(ask_whiskers("Can cats taste sweet things?"))   # 🐱 Nope!
print(ask_whiskers("How long do cats typically live?"))  # 🐱 12–18 years
